# Bounded AST shadow review reporting and dashboard (re-review)

The corrected Phase 4 change adds a local AST-shadow report CLI, a bounded audit-log read model, and a read-only dashboard view. The CLI now requires an explicit UTC window, rejects windows over 31 days before checking the database, applies configurable pilot gates (defaulting to 7 days, 100 records, and 20 typed-capability records), and emits deterministic JSON or Markdown with digest-only event data. The audit boundary validates and normalizes the shadow contract, while the runtime parser is integrated as a production dependency without changing the authoritative policy or database path. **Watch for:** **confirmed** parser-version drift can still receive `clean_review`; **confirmed** the report can count a safety signal while omitting every digest sample for it when that signal falls outside the bounded recent-event source.

## Resolution status

This review is a research snapshot captured before the latest AST-authoritative integration. The two issues above are now covered by the current implementation and regression tests: parser-version drift produces `review_required`, and safety events are selected independently from the bounded clean-event buffer. Keep the findings below as historical rationale; the current source and test suite are authoritative for the resolved behavior.

## High-level view

The CLI contract is now explicit and fail-fast: timestamp syntax, calendar validity, ordering, and the 31-day maximum are checked before `statSync` or `createAuditLog`, and the configured pilot thresholds are echoed in the result. The report distinguishes insufficient evidence from a clean result and includes the status, counts, safety signals, integrity, and thresholds in both output formats.

The audit read model is allowlisted, half-open, UTC-windowed, and bounded for recent observations. Shadow events persist only normalized facts and a digest, and the report/dashboard paths omit correlation IDs and raw request data. **confirmed** the bounded source is an all-event recent buffer, so a safety event outside that buffer remains visible only as an aggregate signal and may have no digest sample for triage.

Shadow observation remains non-enforcing and failure-isolated, and the report script does not start the MCP server or write audit records. **confirmed** the status calculation does not apply the parser-version consistency gate even though it exposes the version distribution, so a mixed-version collection can be presented as clean.

### Report status can be clean with parser-version drift

**confirmed** `createReport` in `bin/ast-shadow-report.mjs:90-93` derives status from data sufficiency, the three safety counters, and integrity; it never checks `observedParserVersionCounts`. The report can therefore contain both parser 16 and parser 17 observations, satisfy the 7-day/100-record/20-typed-record pilot gates, and return `clean_review`. That conflicts with the operational gate requiring one approved parser version in live collection. The action is to add a parser-consistency predicate—normally exactly one observed supported version, or an explicitly configured approved set—and make a violation produce `review_required` with a machine-readable signal and Markdown explanation. The provisional numeric thresholds should remain configurable pilot gates rather than being replaced.

The new tests cover version distributions as aggregate output and cover safety signals individually, but **confirmed** they do not exercise a threshold-qualified mixed-version report. Add that regression case alongside the status change so future parser upgrades cannot silently pass the report gate.

### Safety signals can be detached from their digest samples

**confirmed** `src/auditLog.mjs:563-615` retains only `recentLimit` events while iterating the full window, and `bin/ast-shadow-report.mjs:93-101` then filters that bounded all-event buffer for safety classifications. The CLI deliberately requests the maximum 100 source events at `bin/ast-shadow-report.mjs:227`, which avoids an unbounded sample but does not guarantee that a safety event is represented. With one safety event followed by 100 ordinary events, the aggregate widening/parse/unsupported count is nonzero while `safetyRelevantEvents` is empty. This makes the report insufficient for digest-based triage even though it correctly signals that review is required.

Keep the source bounded, but maintain a separate bounded recent safety buffer during summary construction or issue a bounded safety-only selection. Apply `--recent-limit` to that safety buffer after selection. Add a regression test where the safety event is older than the general 100-event buffer and assert that the report still carries its digest, source, mode, parser, parse status, reason, and classification without exposing correlation or SQL text.
